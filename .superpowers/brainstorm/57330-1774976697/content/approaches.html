Three Architectural Approaches

How the app should work under the hood. Each has different trade-offs for quality, complexity, and flexibility.

A

SVG Pipeline — Recommended

Fetch vector map data (OpenStreetMap via Overpass API), convert to SVG layers, apply CSS-driven styles and transforms, export to high-res PNG/PDF via headless rendering.

Vector = infinite resolution CSS styles = easy presets Clean layer separation
OSM data can be messy Large cities = lots of SVG paths
B

Mapbox/MapLibre Styled Tiles

Use MapLibre GL JS with custom style JSONs for each preset. The map library handles rendering, you customize the style. Export by capturing the canvas at high resolution.

Battle-tested rendering Handles any zoom/location Fastest to prototype
Limited to what map styles can do Hard to add masks/shapes/overlays Canvas export has DPI ceiling
C

Hybrid: MapLibre for Data + Canvas/SVG for Art

Use MapLibre to fetch and display the map for selection/framing, but once the user "captures" a view, extract the vector data and switch to a separate SVG/Canvas rendering pipeline for the poster composition. Best of both worlds.

Familiar map UX for selection Full creative control for output Print-quality vector export
Two rendering systems to maintain Data extraction step adds complexity

My recommendation: A (SVG Pipeline)

Option B (MapLibre styles) is the fastest to prototype, but you'll hit a ceiling fast — masks, shapes, typography overlays, and grain textures all live outside what a map style can do. You'd end up building a compositing layer on top anyway. Option C is smart but adds complexity you don't need yet — MapLibre's interactive map is nice for framing, but a simple search + bounding box gets you there too. The SVG pipeline gives you full control from day one, clean layer separation, and infinite resolution for print. The trade-off is you need to wrangle OSM data into clean paths, but that's a solved problem.
